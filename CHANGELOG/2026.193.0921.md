# 2026.193.0921 — roster agents run UNRESTRICTED; skill-access-by-path; the two-gate merge-authorization model

Fixes THREE independent things (do not conflate them):

**(1) Tools — retire the restricted-tools concept (Candyboxing).** A whitelisted `tools:` set is
the CLAUDE.md "Candyboxing" anti-pattern ("never secure by whitelisting commands; trust the walls,
not the tools; the agent gets the ENTIRE candy store"). **Every roster agent now OMITS the `tools:`
field** → inherits ALL tools (documented Claude Code behavior; equivalent to the built-in
`general-purpose` agent, `Tools: *`). This supersedes the piecemeal `Skill`/`SendMessage`/`Write`
grants — omission grants all at once. (`tools: "*"` is NOT valid and yields ZERO tools — omit the
field.)

**(2) Skill access — `Read` SKILL.md BY PATH is the reliable method (PROVEN); `Skill(name)` is
unreliable regardless of tools.** Invoking a charly-* skill by name depends on it being REGISTERED
in that sub-agent's SESSION — which is INDEPENDENT of the `tools:` field and usually ABSENT.
**Verified live this session:** an unrestricted `Tools: *` validator got `Unknown skill:
charly-internals:git-workflow`, while `Read`ing `plugins/<family>/skills/<name>/SKILL.md` worked
every time (proven across multiple validator runs). So unrestricting does NOT fix skill-by-name;
the reliable method is the file `Read`, and a `Skill(name)` failure is EXPECTED for a sub-agent,
never "skills unavailable".

- **`internals/agents/*.md`** (all 6: check-bed-runner, deploy-verifier, layer-validator,
  pr-validator, root-cause-analyzer, testing-validator) — the `tools:` line is DELETED.
- **`internals/agents/pr-validator.md`** — the "R0" section: `Read` the SKILL.md by PATH (the
  reliable method, with the concrete paths); `Skill(name)` is an optional fast-path and a failure is
  expected.
- **`internals/skills/agents/SKILL.md`** — the roster section's "narrowed `tools:` = role-lane
  discipline" claim is replaced with the unrestricted rule (role = PROMPT + spec, never a tool
  whitelist); "Sub-agent operational invariant #4" separates the two facts (unrestrict for tools;
  Read-by-path for reliable skill access).

**(3) Landing — the TWO-GATE merge-authorization model (corrects a disproven claim).** The
prior `git-workflow` B5 wording said a superproject-rooted validator "posts `success` and merges
with **zero denials**" via `permissions.allow` — DISPROVEN live this session. `permissions.allow`
and the auto-mode classifier are SEPARATE gates: the `Bash(gh api --method POST repos/opencharly:*)`
rule clears the `success` STATUS POST, but the classifier's **Merge-Without-Review** soft-block fires
on `gh pr merge` and is NOT cleared by `permissions.allow` (the merge was denied for BOTH a
superproject-rooted sub-agent AND the main session). The classifier also explicitly rejects a
CLAUDE.md authorization as "manufactured classifier-steering intent," and rejects a compaction-eroded
in-session consent. The ONLY durable, compaction-proof grant that clears the merge gate is an
`autoMode.allow` rule in **USER or MANAGED settings** (never a committed repo `settings.json`, which
the classifier ignores for `autoMode`); a HUMAN must apply it (an agent editing its own permission
config trips the un-bypassable Self-Modification block).

- **`internals/skills/git-workflow/SKILL.md`** — B5 rewritten to the two-gate model (status-post vs
  merge gate), what clears Merge-Without-Review (autoMode.allow > fresh intent > NOT CLAUDE.md), the
  compaction-durability ranking, and the agent-cannot-self-grant guarantee.
- **`internals/agents/pr-validator.md`** — Phase-3 merge step: on a Merge-Without-Review denial the
  validator has already posted the green status + finalized CalVer, so it records the ready-to-merge
  command and STOPS (no retry — a reshaped retry is a bypass); the operator/main-session completes it.
- **`internals/skills/agents/SKILL.md`** — invariant #1: rooting fixes the STATUS POST; the MERGE is
  the separate Merge-Without-Review gate needing the `autoMode.allow` rule.
- The superproject `CLAUDE.md` Post-Execution point 3 carries the matching correction (its own commit).

Rationale: security is the WALL (the disposable target + branch protection + the validator's
fresh-context independence), never a tool whitelist — the CLAUDE.md "Candyboxing" mandate the agent
skill now matches; skill access is made reliable by the deterministic file `Read`; and the landing
docs now match the auto-mode classifier's PROVEN two-gate behavior instead of an aspirational
single-gate claim.
