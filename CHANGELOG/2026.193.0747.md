# 2026.193.0747 — roster agents run UNRESTRICTED (omit `tools:`); the restricted-agent concept is retired

Retires the restricted-tools agent concept — it violated Candyboxing ("never secure by whitelisting
commands; trust the walls, not the tools; inside the box the agent gets the ENTIRE candy store") AND
was the real cause of the recurring "The named skills aren't available as invocable skills in this
environment (not registered in the Skill tool here)" failure. A whitelisted `tools:` set both walls
the agent off from the candy store and gives flaky skill access.

**Fix: every roster agent OMITS the `tools:` field** → inherits ALL tools (documented Claude Code
behavior; equivalent to the built-in `general-purpose` agent which is `Tools: *`). An unrestricted
agent has the `Skill` tool with full plugin-skill access and invokes `charly-internals:git-workflow`
etc. BY NAME — verified live this session (a `Tools: *` teammate loaded `charly-internals:go` +
`git-workflow` by name, full body, no error). This supersedes the piecemeal tool additions (the
earlier `Skill` / `SendMessage` / `Write` grants): omission grants all of them at once. (`tools: "*"`
is NOT valid syntax and yields ZERO tools — omit the field entirely.)

- **`internals/agents/*.md`** (all 6: check-bed-runner, deploy-verifier, layer-validator,
  pr-validator, root-cause-analyzer, testing-validator) — the `tools:` line is DELETED.
- **`internals/agents/pr-validator.md`** — the "R0 — load your PR-validation skills FIRST" section:
  you run UNRESTRICTED, so invoke your skills BY NAME via the `Skill` tool (the SKILL.md files remain
  a universal `Read`-by-path fallback, not the primary method). The Read-by-path-primary band-aid is
  removed.
- **`internals/skills/agents/SKILL.md`** — the roster section's "narrowed `tools:` = role-lane
  discipline" claim is replaced with the unrestricted rule (role = PROMPT + spec, never a tool
  whitelist; the Candyboxing anti-pattern); "Sub-agent operational invariant #4" reframed to
  "unrestricted agents invoke by name; do NOT re-add a `tools:` line to grant a specific tool".

Rationale: security is the WALL (the disposable target + branch protection + the validator's
fresh-context independence), never a tool whitelist — the exact CLAUDE.md "Candyboxing" mandate,
which the agent skill now matches instead of contradicting.
