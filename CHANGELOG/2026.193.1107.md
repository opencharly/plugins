# 2026.193.1107 — D1: the proven concurrent-landing model (per-worktree binaries, overlapping bed gates, delta re-gates) + agent lifecycle hygiene

Documentation-only cutover (D1 of the finish-the-core-minimization program). Codifies
mechanisms PROVEN live on 2026-07-12 (spikes S0 + S0b) plus one operator-mandated
operational rule — immediately, per the standing "constantly codify proven insights"
directive, so the knowledge survives context compaction.

**`internals/skills/agents/SKILL.md`:**
- The bed-scoped intro no longer claims the bed "replaces the git worktree" — the bed is
  the unit WITHIN one source tree; across CUTOVERS the git worktree with its own built
  binary is the gate-isolation unit (the two compose).
- New item **4e — per-worktree binaries (PROVEN, spikes S0+S0b)**: the freshness guard is
  `os.Executable()`-vs-cwd-walked-source-root, so `task build:binary` + `PATH=$PWD/bin:$PATH`
  runs the FULL `charly check run <bed>` R10 sequence from a worktree without ever
  installing; proven concurrently across two worktrees on different branches (two distinct
  beds, overlapping windows, separate `.check`/`.build`, zero cross-contamination, zero
  leftovers, `/usr/bin/charly` untouched). Requirements (sdk + pkg/arch submodules; never
  `task build:charly` from a worktree) and the cross-gate scheduling rule (same-bed-name
  mutex; distinct beds collision-free by construction; one store-lock ceiling + token
  chains globally).
- New section **"Agent lifecycle hygiene — stop what you spawned"**: TaskStop every agent
  the moment its work is accepted; stale finished agents leak tmux panes until spawning
  fails; the fix is cleanup (sweep + kill idle agent panes), NEVER switching away from tmux
  teammate mode — panes are the operator's oversight, and `teammateMode` is session-start-
  snapshotted anyway. (Motivating incident: 17 leaked panes in one session.)

**`internals/skills/git-workflow/SKILL.md`:**
- New B1 subsection **"Concurrent landings"**: with N cutovers in flight, only the merge
  instants (git-inherent) and the real dependency DAG are ordered — implementation,
  validation (concurrent fresh validators), and bed gates all overlap; after each merge,
  `gh pr update-branch` + a risk-proportional DELTA RE-GATE (file-overlap-keyed: empty →
  rebuild+test+lint+status re-post; non-empty → + primary beds; full roster only on
  risky-path overlap). Records the TECHNICAL reason `strict: true` is kept (one shared Go
  package, no CI on main → semantic-conflict blind spot), so the requirement is never
  mistaken for doctrine.
