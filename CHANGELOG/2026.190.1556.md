# 2026.190.1556 — hooks-doctrine: a heredoc commit message is blocked, not scanned

Paired documentation sweep for the superproject cutover that rewrites the PreToolUse
gates onto a shared `shlex` tokenizer with **no fallback** (`.claude/hooks/gitcmd.py` +
`pre-commit-gate.sh` + `pre-push-gate.sh`).

The `/charly-internals:agents` hooks-doctrine previously said the gate's absent-trailer
check treats "a `-F`/heredoc/command-substituted message" as *not scanned for absence,
tier legality still applies". Under the new tokenizer that is no longer true for a
**heredoc**: the gate cannot tokenize a heredoc (or any unbalanced/unquoted quote), so
it **fails closed and blocks** the command — put a long message in a file and use
`git commit -F <file>`. A `-F <file>` or command-substituted message still parses and is
simply not scanned for absence.

The sentence is corrected to state that (R1: documentation divergence is an incident;
R5: swept in the same cutover as the behaviour change). No other doctrine claim referenced
the old regex/heredoc-scanning behaviour.
