# 2026.191.1101 — cloud-init-renderer skill: the seed ISO label is `CIDATA`, and it was never FAT

Claim-keyed sweep (R1/R5) riding the sdk cutover that made the NoCloud seed ISO's volume
identifier ISO 9660-compliant.

The `/charly-internals:cloud-init-renderer` skill carried two false claims, in two places:

- **"a FAT-labeled `cidata` ISO"** — it is an ISO 9660 image (xorriso `-as mkisofs`
  `-joliet -rock`), never FAT. The confusion is understandable: cloud-init's NoCloud
  datasource searches `TYPE=vfat` *and* `TYPE=iso9660`, but charly only ever produced the
  latter.
- **the label is `cidata`** — it is now `CIDATA`. ISO 9660 / ECMA 119 d-characters are
  `A-Z 0-9 _` only, so the lowercase label made xorriso warn on every VM boot
  (`-volid text does not comply to ISO 9660 / ECMA 119 rules`). cloud-init still finds it:
  its NoCloud datasource searches both `LABEL=<fs_label>.upper()` and `.lower()`, with
  `fs_label` defaulting to `cidata`.

The skill now names the shared source (`vmshared.cloudInitVolumeID`) and states *why* the
identifier must be uppercase, so the constraint travels with the claim.

Producer: `opencharly/sdk` — `sdk/vmshared/cloud_init_iso.go` replaced three `-volid cidata`
literals with one shared `cloudInitVolumeID = "CIDATA"` constant.
