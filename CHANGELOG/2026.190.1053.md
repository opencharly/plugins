# 2026.190.1053 — git-workflow: correct the self-approval + autoMode.allow claims

Two claims in the B5 landing section were false. Both are corrected here.

**1. `autoMode.allow` does not take permission-rule syntax.** The prior text told
authors to put `Bash(gh pr merge:*)` into `autoMode.allow` "so the pr-validator
merges autonomously". That is a category error: `permissions.allow` holds deterministic
`Tool(pattern)` RULES, while `autoMode.{allow,soft_deny,hard_deny}` holds the auto-mode
CLASSIFIER's NATURAL-LANGUAGE policy statements (`claude auto-mode defaults`: `allow` has
16 entries, ZERO containing `Bash(`; the settings schema calls them "Rules for the auto
mode classifier … section"). Permission syntax placed there is inert — with such entries
live, a fresh `pr-validator`'s status POST was denied twice.

**2. A commit status is agent-ATTESTED validation, not two-party review.** The prior text
presented "the COMMIT STATUS has no self-approval restriction" as the design's
justification. The classifier's **Self-Approval** rule names this pattern directly:
"triggering a pipeline that marks the agent's own PR's required checks as passed …
regardless of whether the agent believes it verified its own code", and a sub-agent the
session spawned is "an automation the agent controls". A fresh `pr-validator` gives
CONTEXT independence (it catches real defects) but never PARTY independence: same
principal, same token, `required_approving_review_count: 0`.

The section now states the real mechanism: Self-Approval and Merge Without Review are
`[named+specifics]` SOFT blocks (the schema: "actions that user intent can clear"),
clearable only by consent that NAMES the action; a `<teammate-message>` never meets that
bar; a standing operator rule in the SUPERPROJECT's `autoMode.allow` does. The
`gh pr merge --auto` carve-out is stated with its real precondition (required-reviews
branch protection), which these repos do not meet.

**Claim-keyed sibling sweep (R1/R5).** The same false claim lived in
`internals/agents/pr-validator.md`, which told the validator its status POST was
"deterministically permitted (no per-PR classifier judgement)" because
`permissions.allow` carries the POST rule. Refuted by the record: that rule was already
live when a fresh pr-validator's POST was denied, which is why an operator-authored
`autoMode.allow` rule was needed. The agent spec now states that a pr-validator is a
SUB-AGENT whose every action the classifier evaluates, and that only the operator's
standing rule — never a `<teammate-message>` — clears Self-Approval.

**Asymmetry now documented:** posting a `failure` status never trips Self-Approval (it
marks nothing passed), so a FAIL verdict always goes through; only `success` is gated.
