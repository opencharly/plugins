**`internals/skills/agents/SKILL.md`:**

- New section **"The default multi-agent execution model — a most-capable orchestrator driving
  cost-scaled teammates"**. For substantial / multi-cutover work the DEFAULT topology is a
  persistent ORCHESTRATOR coordinating parallel TEAMMATES (one per in-flight cutover) + fresh
  `pr-validator`s (one per ready PR); solo/sequential is the EXCEPTION, for trivial work. Four
  subsections:
  - **Model-tier split** — the MOST-CAPABLE (most-expensive) model runs as the ×1 orchestrator
    (premium paid ONCE, put where the leverage is highest: routing, merge sequencing, and the
    independent RDD-verification of every teammate decision), while a LESS-EXPENSIVE model runs as
    the ×N teammates/validators (each a bounded unit under that verification, scaling the parallel
    bulk affordably). States the PRINCIPLE + cost RATIO, not exact prices; current instantiation is
    a Fable-5 orchestrator (~2× the teammate model's per-token cost) driving Opus-4.8
    teammates/validators.
  - **Maximum parallelization is the DEFAULT** — cutovers implemented concurrently (one teammate
    each) and PRs validated concurrently (one fresh validator each), first-ready-first-merged,
    partitioned by the existing bed-scoped contract.
  - **Concurrent landing** — LINKS to `/charly-internals:git-workflow` "Concurrent landings — N
    open cutovers do NOT serialize beyond git itself" (the `strict: true` + `gh pr update-branch` +
    risk-proportional delta re-gate); does not restate it.
  - **The orchestrator RDDs every decision — bidirectionally** — the orchestrator independently
    re-derives every teammate decision (scope / mechanism / coverage) before accepting it, and
    teammates likewise correct the orchestrator; present-tense principle with the concrete failure
    classes a one-way audit misses (a shared-domain collision surfacing only under simultaneity,
    under-scoped bed rosters, drop-vs-nest coverage gaps, stale rebase bases).
- **Item 4e (Bed-scoped section) extended** — a worktree running the ROOT disposable check roster
  needs `sdk` + `pkg/arch` inited but NOT the `box/<distro>` submodules: root `charly.yml` imports
  the distro namespaces (arch / cachyos / fedora) via remote `@github.com/opencharly/distro-*` refs
  (resolver-fetched into cache, pinned independently of the submodule pointers), so the root check
  beds resolve from local `candy/` layers + registry base images + the remote-fetch cache, never the
  box/* working trees. A `box/<distro>`'s OWN in-submodule beds still need that submodule inited.
