# PR-only agent-validated landing — the pr-validator agent + skill alignment

Part of the project-wide cutover to a PR-only, agent-validated landing model (a
direct push to `main` is forbidden and mechanically disabled; every change lands
through a PR that a FRESH `pr-validator` agent validates and merges). This is the
`plugins` half — all documentation (agent prompt + skills):

- **New agent `internals/agents/pr-validator.md`** — the fresh PR evaluator: given
  a PR, it independently re-validates against R0–R10 + the relevant skills, posts
  the `charly/claude-validation` commit status, and ONLY on PASS generates the
  merge-time CalVer, rewrites the version surfaces on the feat branch, merges
  (`gh pr merge --rebase`), and tags. Encodes the spike-proven sequence
  (authoritative head SHA via `git ls-remote`; `BEHIND` recovery via
  `gh pr update-branch`, never force-push).
- **`internals/skills/git-workflow`** rewritten for the two-step PR flow: the
  no-direct-push-to-main invariant, B1 (author opens PR → fresh evaluator merges),
  B5 (PR path is the SOLE path — the old write-access direct ff-merge "Mode 1"
  deleted), B2/B6/B7 (per-repo two-step landing in dependency order), and CalVer
  generated-and-rewritten by the evaluator at merge.
- **`internals/skills/agents`** — `pr-validator` added to the enforcer roster; the
  hooks-doctrine now names the pre-push main-destination block; the agent-team
  lead opens the PR (a fresh evaluator merges).
- **`internals/skills/skills`, `internals/skills/cutover-policy`,
  `build/skills/migrate`, `internals/agents/testing-validator`** aligned: CalVer
  is per-merge (not per-push), the schema version is finalized by the evaluator at
  merge, and the mirroring registry + agent count updated.

Documentation-only (agent prompt + skill prose); the runtime enforcement (the
`pre-push-gate` main-block, the branch-protection helper, and the applied GitHub
branch protection) lands in the superproject half.
