# 2026.191.0428 — deploy-into nesting, the three remote surfaces, and four measured traps

Findings from an RDD spike and a failed R10 roster, each verified against the live system
before being written here. Two of them are R1 documentation divergences: a skill asserted a
duration off by 7×, and two skills implied a skipped bed is a passed bed.

**1. Deploy-into nesting had no worked example — and that cost real time.**

A `local:` node placed **under** a `vm:`/`pod:` node deploys **INTO** that venue. Nesting is
tree position; the child carries **no `host:` field**, and `kit.NestedExecutor{Parent, Jump}`
composes onto the parent's venue (`nested:<jump>/ssh://arch@…`). The mechanism was mentioned
only parenthetically, so it was easy to miss and easy to reinvent.

`/charly-core:deploy` gains a **"Deploy-into nesting"** section with the canonical shipped
example — `check-arch-vm` → `arch-host` (`box/arch/charly.yml`), whose checks assert the
install ledger at `/home/arch/.config/opencharly/installed/deploys`, i.e. **in the guest** —
and a table distinguishing the three ways a `local:` deploy reaches a machine.

**This is how to run `local:`-target testing inside a VM** — it keeps the `local:` authoring
shape and confines every write to the guest.

**And the skills now state what it COSTS.** `ShellExecutor` runs only for a TOP-LEVEL `local:`
deploy with `host: local` (or absent). **All three remote surfaces bypass it, nesting
included**: a nested child runs on `NestedExecutor` over the parent venue, and
`RootExecutorForDeployNode` "does NOT handle the nested-inside-a-parent case"
(`sdk/deploykit/deploy_chain.go`). Relocating a `host: local` bed — by nesting, a `host:`
retarget, or `charly --host` — deletes its `ShellExecutor` + `HostDeployTarget` coverage.
`check-local` calls itself the "sole proof" of that path; the coverage cannot ride along, it
must be replaced deliberately.

A first draft of this cutover claimed nesting preserved that coverage; a fresh `pr-validator`
refuted it from the code. The *correction* then over-generalized — asserting that "all three
rows" of `/charly-core:deploy`'s table bypass `ShellExecutor`, when that table's FIRST row IS
the `ShellExecutor` case (the sentence had been transposed from `/charly-local:local-deploy`,
whose table lists only the three REMOTE surfaces). The validator caught that too. Both
refutations are recorded here rather than quietly reworded: a skill that contradicts its own
adjacent table is the same defect class this entry exists to remove.

**2. The three remote surfaces are distinct, and only one was documented in a skill.**

`/charly-local:local-deploy` gains a table separating them:

| Mechanism | Executor | Where the work happens |
|---|---|---|
| `local: {host: user@machine}` — a deploy FIELD | `SSHExecutor` | that machine, driven from here |
| a `local:` node nested under `vm:`/`pod:` — tree position | `NestedExecutor` | inside the enclosing deployment |
| `charly --host <alias> <verb>` — a global CLI FLAG | n/a | a fresh `charly` runs remotely |

`charly --host` (`charly/host_exec.go`) shells out to `ssh <target> charly <argv>`. Aliases
resolve via `charly settings set hosts.<alias>`, else fall through to `ssh(1)` — which is why
the `charly-<vmname>` stanza `charly vm create` writes just works. `settings`/`version`/`ssh`
are `LocalOnly`. **`buildRemoteArgv` strips `--host`, `--dir`/`-C`, and `--repo`**, so the
remote charly must find its project via its own cwd or `CHARLY_PROJECT_DIR`/`CHARLY_PROJECT_REPO`.
`ssh host cmd` is non-interactive (no `/etc/profile`, no `~/.bashrc`); `/etc/environment` DOES
reach it via `pam_env` — verified on a live guest.

**3. Four measured traps, each reproduced before documenting.**

- **`summary.yml`'s `ok:` does not distinguish a PASS from a PREREQ SKIP.** A skipped bed
  (exit code 3) writes `ok: true`, `total_seconds: 0`, and one `prereq-*-skipped` step. The
  exit code is the only discriminator. Corrected in `/charly-check:check` **and** in
  `/charly-internals:agents` (claim-keyed sweep — both said `ok:` was the verdict).
- **`charly vm destroy` takes the `kind: vm` ENTITY name, not the bed name**, and **exits 0
  printing `Destroyed VM …` for a domain that does not exist** — a silent no-op success that
  leaves an orphan running. `charly vm destroy definitely-not-a-vm-xyz` → `exit=0`. Confirm
  with `charly vm list`. Recorded in `/charly-vm:vm` and `/charly-check:check`.
- **`charly bundle add vm:<vm> <localpkg-candy>` exits 0 WITHOUT installing.** It prints
  `guest charly absent/outdated; host charly provided at /tmp/charly-<ver> for deploy use` —
  meaning the deploy STAGED the host binary for its own use. The `localpkg:` build needs the
  bed runner's automatic `--dev-local-pkg`. Do not read rc=0 as "installed". Recorded in
  `/charly-vm:vm`.
- **`task build:charly` rewrites the TRACKED `pkg/arch/PKGBUILD` `pkgver=` stamp**, leaving
  the submodule (and via the gitlink, the superproject) dirty. Reproduced:
  `pkgver=2026.185.0752 → 2026.190.2230`. Recorded in `/charly-internals:go`.

**4. Bed wall-clock figures are load-dependent, and the idle numbers mislead.**

`/charly-check:check` said `check-pod` takes ~110s. Its newest `summary.yml` records
**`total_seconds: 842`** under a concurrent roster. The table now gives idle *and* under-load
figures (`check-openclaw-full-pod` 1414s, `check-sway-browser-vnc-pod` ~2477s) and states that
a bed's own newest `summary.yml` is the only honest number — which is exactly what
`/verify-beds` reads to decide whether a sub-agent may own it (≥600s ⇒ deferred).
