# 2026.193.1615 — Unify per-merge tagging AND the CHANGELOG rule: `plugins` and `pkg/*` are full first-class repos, no exemptions (sdk's scheme is a Go-module requirement)

Documentation-only cutover (operator directive: every repo uses the same rules,
methods, and CalVer versioning — no repo-level exemptions). Two unified themes:
(1) removes the `plugins` / `pkg/*` TAG-EXEMPTION from every skill + agent surface
and reframes the one remaining exception (`sdk`) as a HARD Go-module technical
requirement rather than an arbitrary exemption; (2) states, present-tense, that
`plugins` is no CHANGELOG-exception either — EVERY `plugins` landing carries a
`CHANGELOG/<YYYY.DDD.HHMM>.md` entry (author writes a placeholder, the `pr-validator`
finalizes the CalVer at merge, `git mv`-ing it to the merge stamp), and now that
`plugins` is tagged the one-stamp-for-both invariant (the SAME CalVer names the
changelog file AND the tag) applies to `plugins` too.

**New standing rule (stated consistently on every surface):** every repo — the
superproject, every `box/<distro>`, `plugins`, and `pkg/*` — is tagged
`v<YYYY.DDD.HHMM>` on its own merged HEAD by the `pr-validator` at merge. The tag
marks the MERGE (decoupled from any `charly.yml` `version:` schema field), so a repo
needs no `charly.yml` to be tagged. `sdk` ALONE uses `v0.<YYYYDDD>.<HHMM
leading-zeros-stripped>` because a Go-module version is semver: a leading-zero HHMM
segment is forbidden, and a `major ≥ 2` would force a `/vN` module-path suffix that
breaks every `import github.com/opencharly/sdk` — a technical constraint, not an
exemption. Tagging `plugins` / `pkg/*` is harmless: only a SUPERPROJECT `v*` tag
triggers `release-packages.yml`; a `plugins` / `pkg/*` tag fires no workflow.
