# 2026.192.0844 — sdk tag scheme: strip ALL leading zeros (unambiguous)

Fix a documentation divergence that produced an invalid sdk release tag during the P6 landing.

`pr-validator.md` and two `git-workflow` statements described the sdk Go-module tag as
`v0.<YYYYDDD>.<HHMM>` / `<HHMM stripped>`, omitting the explicit "leading-zeros-stripped" rule that
`git-workflow` (step 0), `migrate`, and `plugin` already carried. A fresh `pr-validator` followed the
under-specified `pr-validator.md:409` wording and tagged a morning-time (07:33) sdk merge
`v0.2026192.0733` — a leading-zero patch segment is INVALID semver, which makes every consumer's
`go.mod` unparseable in module mode (`go build` / `pluginsgen` fail).

- **`pr-validator.md` (Phase 3 CalVer step)** now states the sdk tag as
  `v0.<YYYYDDD>.<HHMM with ALL leading zeros stripped>` with the worked examples (`0751`→`751`,
  `0733`→`733`, `0009`→`9`) AND the exact shell to compute it
  (`SDK_MIN=$(echo "$HHMM" | sed 's/^0*//'); SDK_MIN=${SDK_MIN:-0}`), and calls out that a leading-zero
  sdk tag is invalid, not cosmetic.
- **`git-workflow` (Non-negotiable invariants + CalVer section)** carry the same explicit
  leading-zeros-stripped wording, so all four tag-scheme statements agree.

Claim-keyed R5 sweep: every current-state surface stating the sdk tag scheme now carries the
leading-zeros-stripped rule; only `CHANGELOG/` history retains the old bare form. The canonical
`v0.2026192.733` tag was added (add-only) at the P6 sdk merge commit to unblock the build.
