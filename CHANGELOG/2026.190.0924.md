# 2026.190.0924 — git-workflow: pr-validator merge authority needs BOTH permission layers

Corrects the B5 "`gh pr merge` allow-rule" paragraph. It claimed
`permissions.allow` alone makes the auto-mode classifier not re-litigate the merge
— true for the interactive MAIN session, but INCOMPLETE for a sub-agent. A fresh
`pr-validator` is a SUB-AGENT and runs under AUTO MODE, which consults a SEPARATE
`autoMode.allow` list; a command in `permissions.allow` but not `autoMode.allow`
falls to the auto-mode classifier's non-deterministic judgement and silently DENIES
an already-validated sub-agent merge ("teammate directive is not user intent").

The paragraph now states the load-bearing rule: `Bash(gh pr merge:*)` (and the
`gh api --method POST` status post) MUST be in BOTH `permissions.allow` AND
`autoMode.allow` (`["$defaults", …]` to preserve the defaults) for the pr-validator
to merge autonomously. The paired `.claude/settings.json` change lands in the
superproject. Server-side branch protection remains the real gate — an unvalidated
PR still cannot merge regardless.
