# 2026.197.0700 — Codify charly-binary discipline for multi-teammate / multi-worktree setups

Documentation-only cutover. The guidance for which `charly` binary a teammate
should use during in-flight multi-agent work existed only as scattered
fragments — the go skill's `task build:charly` Quick Reference row, the
agents skill's four `build:binary`/freshness-guard mentions (the B3-mirroring
"Implementation workflows are bed-scoped too" barrier bullet and the 4e
"Per-worktree binaries" item), and operator session memory — with no single
owning section a teammate could be pointed at.

## The motivating incident

A teammate's worktree smoke bed (`check-commands-local`, a HOST-LOCAL bed
whose `command:` steps shell out to bare `$PATH` `charly`) tripped the
stale-binary freshness guard. The teammate "fixed" it by running
`task build:charly` — a HOST install of `/usr/bin/charly` from an UNMERGED
branch, violating the per-worktree-binary discipline the agents skill's own
4e item had already proven (2026-07-12, spikes S0+S0b) but never stated as a
general rule with its own name.

## The second, complementary incident

Minutes after the first, a second incident proved the COMPLEMENT of the same
rule: a teammate edited a header comment in its own worktree while its OWN
`check-sidecar-pod` bed run was mid-flight, at step 6. The per-tree freshness
guard tripped on the bed's next step (`check-live`), citing "newer source:
`charly/materialize.go`", and failed an otherwise-green run. Correctly RCA'd
as self-inflicted (not a product defect), the teammate rebuilt via
`task build:binary` and re-ran fresh. Where the first incident showed
"across worktrees, no freeze is needed", this one showed the complementary
boundary: "within ONE worktree, freeze your OWN `charly/*.go` while your OWN
bed is mid-run" — the guard checks at EVERY heavy verb, not just at the
start.

## The fix (round 1)

Added ONE new authoritative section, "The charly binary in a multi-teammate /
multi-worktree setup", to `internals/skills/agents/SKILL.md` (the sole owner
of multi-agent discipline per the Authoritative-copy registry). It covers:
two binaries and their two roles; the host-install boundary; the
stale-binary guard's per-tree semantics (`charly/main_freshness.go`); why a
host-local bed can never gate worktree work; the multi-worktree concurrency
corollary (no freeze barrier needed ACROSS distinct worktrees, unlike the
shared-tree B3 model); the complementary within-worktree self-freeze rule
(freeze your OWN `charly/*.go` for the duration of your OWN bed run — the two
scopes are complementary, not the same rule, and neither replaces the other);
a pointer to the side-effects already documented in the go skill; and the
post-merge resync procedure.

Added clarifying cross-notes (not restatements) at the THREE places that
already touched this territory, so the shared-tree freeze rule and the new
multi-worktree rule can never be conflated again:
- `internals/skills/git-workflow/SKILL.md` B3's "Freeze `charly/*.go` during
  the bed phase" bullet.
- `internals/skills/agents/SKILL.md`'s own restatement of that same barrier,
  in "Implementation workflows are bed-scoped too".
- `internals/skills/agents/SKILL.md`'s 4e "Per-worktree binaries" item, which
  now points to the new canonical section for the general rule and keeps
  only its own concurrent-bed-proof + CalVer-stamping detail.
- `internals/skills/go/SKILL.md`'s Quick Reference table gets a one-line
  pointer to the new section (not a restatement).

## Round 2 — the policy itself changed: `task build:charly` is REMOVED

An operator directive (2026-07-16) landed WHILE round 1 was in flight: the
per-worktree `task build:binary` workflow becomes the ONE AND ONLY
development approach, and the system-wide install machinery (the former
`charly`/`install`/`install-arch` Taskfile tasks, which built AND installed
in one step) is DELETED entirely in a paired superproject cutover. The host
`charly` becomes a distro-native package (`.pkg.tar.zst`/`.rpm`/`.deb`, built
containerized via `task pkg:arch`/`pkg:fedora`/`pkg:debian`/`pkg:all` +
`charly box pkg`, or downloaded as a release), installed with the distro's
OWN package manager — never via a Taskfile install step. `task
install-portable` (a portable `$HOME/.local/bin/charly` copy) is explicitly
KEPT for solo bootstrap, but — since it writes to a shared `$HOME` location
exactly like a package install — it is equally off-limits as a
multi-teammate in-flight dev-loop shortcut, not a loophole in the boundary.

Round 1's own new section, and every OTHER genuine `task build:charly` /
`task build:install` dev-workflow mention this repo's `git grep` found
(re-verified independently, not taken from the operator's own file-count
estimate), were rewritten to the new policy — 10 files, ~30 hunks:
`internals/skills/agents/SKILL.md` (the round-1 section itself, re-edited),
`internals/skills/git-workflow/SKILL.md` (B3's barrier rule + a landing
gotcha describing a PKGBUILD side-effect that no longer occurs, REMOVED —
confirmed by reading `charly box pkg`'s implementation: it never touches the
tracked `pkg/arch/PKGBUILD`), `internals/skills/go/SKILL.md` (the Quick
Reference table gains a "Package" row and drops the deleted "Install" row;
five other dev-loop mentions), `internals/skills/go-quality/SKILL.md`,
`internals/skills/plugin/SKILL.md` (the `pluginsgen` mention only — its
"build-prep" HostBuild-seam architecture paragraphs, a sibling teammate's
fenced content, were left untouched — confirmed disjoint by reading both
paragraphs before editing), `internals/agents/testing-validator.md`,
`build/skills/build/SKILL.md` (7 hits: cache-invalidation table, Host
Bootstrap, and 3 Troubleshooting sections rewritten to the new install
paths — its own "build-prep" HostBuild-seam paragraph, unrelated content,
also left untouched), `check/skills/check/SKILL.md` (the R10-gate-by-change-

**Two findings surfaced during the sweep, reported rather than silently
resolved:**
- `task build:binary`'s current Taskfile body does NOT run the `pluginsgen`
  regeneration step that the now-deleted `charly` task ran before its `go
  build` — a real gap if `binary` becomes the sole dev build task without
  gaining that step. Documented as `task build:binary` per the operator's
  explicit sweep instruction (the eventual state this cutover pairs with),
  with the gap flagged to the operator/orchestrator as a probable follow-up
  for the paired superproject cutover.
- `task install-portable`'s survival (confirmed against `taskfiles/Build.yml`
  directly, not assumed from the operator's summary) writes to a host
  location precisely like a package install; this cutover's prose scopes the
  "absolute" host-install boundary to in-flight multi-teammate work
  specifically, rather than asserting the Taskfile has zero host-writing
  targets (which would have been false the moment a reader checked).

This plugins-repo leg pairs with a parallel superproject cutover (task-
tracked; sequenced after an unrelated superproject PR) that performs the
actual Taskfile deletion, the `charly/main_freshness.go` remediation-string

and land independently or concurrently — this leg documents the policy the
other leg implements.
