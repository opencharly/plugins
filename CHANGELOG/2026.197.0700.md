# 2026.197.0700 — Codify charly-binary discipline for multi-teammate / multi-worktree setups

Documentation-only cutover. The guidance for which `charly` binary a teammate
should use during in-flight multi-agent work existed only as scattered
fragments — the go skill's `task build:charly` Quick Reference row, the
agents skill's four `build:binary`/freshness-guard mentions (the B3-mirroring
"Implementation workflows are bed-scoped too" barrier bullet and the 4e
"Per-worktree binaries" item), and operator session memory — with no single
owning section a teammate could be pointed at.

## The motivating incident

A teammate's worktree smoke bed (`check-commands-local`, a HOST-LOCAL bed
whose `command:` steps shell out to bare `$PATH` `charly`) tripped the
stale-binary freshness guard. The teammate "fixed" it by running
`task build:charly` — a HOST install of `/usr/bin/charly` from an UNMERGED
branch, violating the per-worktree-binary discipline the agents skill's own
4e item had already proven (2026-07-12, spikes S0+S0b) but never stated as a
general rule with its own name.

## The second, complementary incident

Minutes after the first, a second incident proved the COMPLEMENT of the same
rule: a teammate edited a header comment in its own worktree while its OWN
`check-sidecar-pod` bed run was mid-flight, at step 6. The per-tree freshness
guard tripped on the bed's next step (`check-live`), citing "newer source:
`charly/materialize.go`", and failed an otherwise-green run. Correctly RCA'd
as self-inflicted (not a product defect), the teammate rebuilt via
`task build:binary` and re-ran fresh. Where the first incident showed
"across worktrees, no freeze is needed", this one showed the complementary
boundary: "within ONE worktree, freeze your OWN `charly/*.go` while your OWN
bed is mid-run" — the guard checks at EVERY heavy verb, not just at the
start.

## The fix

Added ONE new authoritative section, "The charly binary in a multi-teammate /
multi-worktree setup", to `internals/skills/agents/SKILL.md` (the sole owner
of multi-agent discipline per the Authoritative-copy registry). It covers:
two binaries and their two roles (`/usr/bin/charly` via `task build:charly`
vs. per-worktree `bin/charly` via `task build:binary`); the host-install
boundary (`task build:charly` forbidden during in-flight work); the
stale-binary guard's per-tree semantics (`charly/main_freshness.go`); why a
host-local bed can never gate worktree work; the multi-worktree concurrency
corollary (no freeze barrier needed ACROSS distinct worktrees, unlike the
shared-tree B3 model); the complementary within-worktree self-freeze rule
(freeze your OWN `charly/*.go` for the duration of your OWN bed run — the two
scopes are complementary, not the same rule, and neither replaces the other);
a pointer to the side-effects already documented in the go skill; and the
post-merge resync procedure.

Added clarifying cross-notes (not restatements) at the THREE places that
already touched this territory, so the shared-tree freeze rule and the new
multi-worktree rule can never be conflated again:
- `internals/skills/git-workflow/SKILL.md` B3's "Freeze `charly/*.go` during
  the bed phase" bullet.
- `internals/skills/agents/SKILL.md`'s own restatement of that same barrier,
  in "Implementation workflows are bed-scoped too".
- `internals/skills/agents/SKILL.md`'s 4e "Per-worktree binaries" item, which
  now points to the new canonical section for the general rule and keeps
  only its own concurrent-bed-proof + CalVer-stamping detail.
- `internals/skills/go/SKILL.md`'s Quick Reference table gets a one-line
  pointer to the new section (not a restatement).

No identifier was deleted or renamed. `internals/skills/plugin/SKILL.md`,
`build/skills/build/SKILL.md`, and `build/skills/generate/SKILL.md` (owned by
a sibling teammate's build-prep docs leg) were swept for the same
conflation and found clean — every `task build:charly` mention there is a
legitimate single-checkout setup/rebuild instruction, not the multi-worktree
claim this cutover corrects, so they were left untouched.
