# 2026.190.1823 — landing doctrine: squash-merge, PR-update-by-default, and the real merge authority

Three corrections to the autonomous-landing doctrine, established by controlled experiment
and by landing, not by inference.

**1. The merge + status-post authority is `permissions.allow` — there is no `autoMode`.**

The skills asserted that a standing natural-language rule in the superproject's
`autoMode.allow` is what clears the classifier's Self-Approval soft block. **That key does
not exist** — not in `.claude/settings.json`, not in `.claude/settings.local.json`, not in
`~/.claude/settings.json`. What actually authorizes the landing is the committed
`permissions.allow` pair:

```json
"Bash(gh pr merge:*)",
"Bash(gh api --method POST repos/opencharly:*)"
```

Proof by landing: fresh, superproject-rooted `pr-validator` agents posted `success` and
merged with **zero denials** on a host carrying no `autoMode` key at all. Two consequences
are now stated where they belong:

- The rules resolve from the agent's PROJECT ROOT (= its working directory). `plugins/` and
  `sdk/` ship no `.claude/`, so a submodule-rooted validator loads **no permission rules**
  and its `success` POST is denied as Self-Approval (*"the only authorization comes from a
  `<teammate-message>`"*). This is the W1 invariant, proven by controlled experiment —
  rooted in `plugins/`: DENIED; rooted in the superproject, same agent, same rules:
  ALLOWED, merged, zero denials. It is the real root cause of every "the rule is inert"
  conclusion drawn earlier.
- A PreToolUse block followed by a **reshaped retry** is flagged as a bypass attempt, after
  which the classifier denies later actions `permissions.allow` would otherwise resolve.
  **Treat any block as a DENIAL; never reshape and retry** (W3). This cost a real landing.

Do not "fix" a permission denial by editing the rule until the agent's project root is
confirmed.

**2. `main` gains exactly ONE commit per cutover — the evaluator merges `--squash`.**

Atomicity is a property of `main`, not of the `feat/` branch. The evaluator composes the
squash message (subject + body + the author's `Assisted-by:` trailer) and merges
`--squash --delete-branch`, folding the author's change, any review-round fix commits, and
the merge-time version stamp into one commit. The flow previously merged `--rebase`, which
landed two commits (change + version stamp) and — because a *fix* commit would have landed
as a third — forced a rewrite-or-recreate choice on every review round.

**3. A PR is UPDATED IN PLACE; closing-and-recreating is forbidden for iteration.**

When a review demands changes, APPEND a fix commit and push it fast-forward: the status
resets and the evaluator re-runs. Closing a PR is reserved for work that will not land at
all (a disproven premise, an abandoned approach). This is what makes the absolute
no-force-push rule livable — a branch carrying five fix commits still lands as one commit.
`git commit --amend` on a PUSHED branch is likewise forbidden: it diverges the branch and
can only be published by a force-push. Amend only before the first push.

**Also encoded (from the same investigation):**

- `/charly-internals:git-workflow` B5 gains **the autonomous-landing contract**: spawn every
  `pr-validator` rooted in the SUPERPROJECT, for a PR in ANY repo; drive submodules with
  literal `git -C /abs/path` + `gh --repo`; never `cd <submodule> &&`. B7 already stated
  this for the commit gate — it is equally load-bearing for PERMISSIONS.
- `/charly-internals:agents` gains **"Sub-agent operational invariants"**: project root =
  cwd; a permission denial ENDS the agent's turn (so verdicts are written durably BEFORE
  any gated action — a `failure` status/comment is never gated); reconnect via durable
  state (statuses, comments, `~/.claude/projects/<cwd-slug>/<uuid>.jsonl`) rather than the
  message channel; wait with a `run_in_background` until-loop whose exit condition covers
  EVERY terminal state, because silence is not success.
- `internals/agents/pr-validator.md` gains operating invariants W1–W3 so every future
  validator inherits the fix: stay in the superproject; record the verdict durably first;
  never route around a denial, and never post `success` on a PR you did not pass.
