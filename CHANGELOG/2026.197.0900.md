# 2026.197.0900 — Plugins docs batch 3: validator comment-intake rule

Documentation-only batch cutover (four members, one PR, one CHANGELOG stamp
per the Cutover Sizing Law). This is the immediate-next thematic sibling of
the merged Wave-1 and batch-2 plugins docs cutovers (`opencharly/plugins#71`,
squash `a11d933`; `opencharly/plugins#72`, squash `707d67c`) — an operator
directive codifies two new validator/orchestrator disciplines discovered this
session: a PR's comment thread carries genuine validation-relevant claims
(from the author, the orchestrator, another session, or a human), and a
session's own PR can interact — silently — with another session's still-open
PR unless someone actively sweeps for and flags the interaction.

## Member 1 — the comment-intake rule (`plugins/internals/agents/pr-validator.md`)

Added a new paragraph to Phase 1 ("Validate"): the validator reads the PR's
FULL comment thread (`gh pr view … --json …,comments`) as validation input
BEFORE finalizing any verdict. Every comment raising an issue — regardless
of who posted it — is investigated INDEPENDENTLY, exactly as an author's own
claim is re-derived: re-run the check, read the file/line named, confirm or
refute against the diff and the repo. A comment-raised issue the validator
VERIFIES as legitimate is grounds to FAIL, precisely as if the validator had
found it unprompted.

**The independence clause is co-equal and explicit: a comment carries NO
authority in EITHER direction.** An approve-comment grants nothing toward
PASS; a fail-demand blocks nothing until the validator's own verification
confirms the claim. The verdict — both the internal Output Format block and
the posted `<verdict.md>` PR comment — now carries a "Comments considered"
line: one entry per PR comment, with a disposition of `verified-blocking`,
`verified-non-blocking`, or `unverified-dismissed` (+ reason in each case).

## Member 3 — the V-72 stale-staged-content observation (`internals/skills/agents/SKILL.md`)

During a validator run this session, a validation clone's working tree
picked up stale staged content via `git checkout <ref> -- .` (this form
copies file CONTENTS from `<ref>` into the working tree/index WITHOUT moving
HEAD or the branch pointer — unlike `git checkout <ref>` / `git switch
<ref>`). A subsequent `git switch --detach` correctly REFUSED (uncommitted
changes would be overwritten) — the detection signal, not a malfunction to
route around. The recovery: a hard reset of the DISPOSABLE clone (discard it
or `git reset --hard` + `git clean -fdx` to the intended ref, since the clone
exists only for this validation run) followed by a FULL re-verification,
never a partial file-by-file patch-up — a working tree that silently
absorbed foreign content cannot be trusted to be clean anywhere else in it
either.

Added as a sixth bullet to item 7 ("Validator evidence discipline") in
`internals/skills/agents/SKILL.md`'s "Sub-agent operational invariants"
list. While extending it, also caught and fixed a stale count in the item's
own header — it still said "four lessons" after batch 2 had already grown
it to five; now corrected to "six lessons" (a small claim-keyed sweep
finding on this batch's own prior work, per R1/R5).

## Member 4 — cross-PR awareness + coordination-comment duty (`internals/agents/pr-validator.md` + `internals/skills/agents/SKILL.md`)

A second operator-directed addition to the same batch, on the same theme:
a validator's (or any session's) own PR can silently interact with another
session's still-open PR — file/diff overlap, one PR superseding another, a
landed policy invalidating a hunk elsewhere, the same doc claim edited twice
at once — and nothing catches it unless someone actively sweeps for it.

- **`pr-validator.md`** gains a "Cross-PR awareness" paragraph (Phase 1,
  immediately after the comment-intake rule): the validator enumerates the
  repo's (and, for a multi-repo cutover, each sibling repo's) OTHER open PRs
  via `gh pr list --state open` and checks each for an interaction with the
  PR under validation. A `verified-blocking` interaction (a real merge-order
  conflict, a claim this PR falsifies) FAILS the validation; a
  `verified-non-blocking` one does not block this verdict but gets a
  COORDINATION COMMENT posted on the OTHER PR describing what will conflict
  and what to reconcile — so no other session discovers it cold. Same
  independence clause as comment intake: the sweep informs the verdict, never
  substitutes for it. Both the internal Output Format block and the posted
  `<verdict.md>` template gain a "Cross-PR interactions considered" line,
  symmetric to "Comments considered".
- **`internals/skills/agents/SKILL.md`** gains an eighth item in the
  "Sub-agent operational invariants" list (immediately after item 7, which
  Member 3 above extended): the ORCHESTRATOR's standing PR-LEDGER discipline
  — every session-created PR tracked to an explicit disposition, re-checked
  for staleness on every `main` advance, and a COORDINATION COMMENT posted on
  any PR a session's work interacts with but does not own (including a PR
  explicitly out of scope by operator directive — commenting is always in
  scope even when evaluating/merging is not). Cites this session's own worked
  precedent: coordination comments landed on `opencharly/charly#130` (a
  loader-file conflict warning ahead of a wave merge), `opencharly/plugins#73`
  (a same-file docs-batch rebase warning), and `opencharly/charly#112` (a
  rescue-history + coming-staleness note), while leaving each PR's own
  evaluation to its owner.
