# 2026.192.1101 — pr-validator: explicit bed-coverage check + re-run authority

Strengthen the fresh `pr-validator`'s R10 evaluation so a runtime-class PR must name the EXACT eval beds it
ran and prove them, and the evaluator independently verifies + re-runs when suspicious.

- **Item 2 (change class → gate)** gains a mandatory BED-COVERAGE CHECK: the evaluator maps the diff to the
  R10-gate-by-change-class matrix and confirms the PR ran EVERY bed that class requires, by name — a
  cross-cutting loader/resolver/IR/fetch/build-spine change → every matching disposable bed CONCURRENTLY on
  ONE binary; a candy/box/deploy change → the bed that COMPOSES the changed entity; a touched substrate →
  that substrate's bed. A gate that ran the wrong, too-few, or cannot-fail-on-the-change beds FAILS, and a
  runtime-class PR that does not list the exact beds + per-bed results is incomplete.
- **Item 11 (R10)** gains NO-benefit-of-the-doubt bed-evidence verification: for EVERY claimed bed the
  evaluator opens `.check/<bed>/<calver>/summary.yml` and confirms it exists, is `ok: true`, and its step
  count / duration / run-calver / binary version match the claim; a concurrent-roster claim must show one
  binary + a coherent launch window. When anything looks off — a missing/contradictory summary, the wrong
  or too-few beds, an inflated tier, or a result it cannot reconcile — the evaluator RE-RUNS the correct
  beds itself (`charly check run <bed>` on the disposable target) and decides on ITS run, never the author's
  word. A suspicious-but-unverified R10 claim is a FAIL until the evaluator's own run confirms it.

Pairs with the org PR template's explicit eval-bed-list requirement (opencharly/.github,
`PULL_REQUEST_TEMPLATE.md`).
