## Member 7 — a new proven lesson: merged `CHANGELOG` files are IMMUTABLE (`internals/skills/git-workflow/SKILL.md`)

A fresh `pr-validator` FAILED an unrelated follow-up PR this session for folding
+60 lines of new narrative into an already-merged, already-tagged
`CHANGELOG/<CalVer>.md` — an understandable reconciliation instinct after a
mid-flight `main` merge (the follow-up PR's author wanted its own related work
documented alongside the original entry), but the wrong direction: a merged
CHANGELOG file is closed history, exactly like the tag sharing its CalVer, and
editing it after the fact would have created a permanent filename↔tag
divergence the moment it landed. The validator caught it before merge.

Added a new paragraph to the "CalVer — generated AND rewritten at MERGE, by the
evaluator" section (right after the existing "ONE fresh stamp per merge,
immutable" paragraph, the natural anchor since it already discusses tag
immutability): a merged `CHANGELOG/<CalVer>.md` is immutable; follow-up work in
the SAME theme, branch, or session never appends to or edits it, and instead
writes its OWN new placeholder entry for its OWN evaluator to stamp at its OWN
merge. This is this batch's own SEVENTH member — landed here rather than as a
solo PR, per the Cutover Sizing Law, since batch 4 is already open and held for

## Member 8 — validator/worktree coordination insight (`internals/skills/agents/SKILL.md`)

A new proven lesson (motivating incident: PR `#133`) added to the "Sub-agent
operational invariants" area, right after "Stop ONLY your own children, by task
id": a validator worktree (`charly-wt-v133`) was cleanly `git worktree remove`d
by the merge/cleanup flow WHILE an agent's self-model still believed it was
mid-validation — the merge had actually completed 79s earlier (confirmed on
live GitHub). Earlier "uncommitted drift" and "a revert with zero reflog ops"
symptoms in the same tree were the SAME collision under a different name: two
actors sharing one worktree path, one's `git checkout --`/`git clean`
clobbering the other's in-flight edits. Codified as a new subsection,
"Worktree lifecycle + validator identity — orchestrator-owned, never assumed",
stating three rules: (1) worktree lifecycle is orchestrator-owned — a
validator/teammate never removes a worktree it doesn't exclusively own, and
every validator gets a UNIQUE worktree path, never a shared per-PR path a
sibling could remove; (2) one validator identity per PR — never re-brief or
re-spawn a validator into re-validating an already-merged PR; (3) the
orchestrator confirms merge state from LIVE GitHub
(`gh pr view <n> --json state,mergeCommit,mergedAt`) before treating any
"still validating" report as actionable — a desynced self-model is not ground
truth. Landed here (rather than the next docs batch, as originally planned)
because this PR was already being re-opened for the trailer-form reconciliation
below — folding it in is free, same repo, same theme, per the Cutover Sizing
Law.
