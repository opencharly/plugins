# Strengthen the forbidden-excuse catalog: SIZE is the same forbidden argument as difficulty

The cutover-policy + strict-policy forbidden-excuse catalogs banned DIFFICULTY framings but
left SIZE-based loopholes an agent could rationalize a stop through. Close them:

- Add the size framings to the forbidden internal-voice / catalog lists: "it's huge", "it's a
  multi-cutover sub-program", "it's a multi-session effort", "it can't be finished this
  session / at a session boundary", "I've scoped it as a focused effort for later", "ready to
  grind it whenever you want".
- State the rule explicitly: **size is the same forbidden argument as difficulty** — a large
  cutover DECOMPOSES into atomic per-unit cutovers, each landed NOW (decomposition is forward
  motion, never a deferral device).
- **Reverting / discarding already-working code because the remaining scope is large is the
  worst form of the excuse** — it destroys progress to manufacture a stopping point.
- The excuse stays forbidden when dressed as "an honest assessment", "a scoping finding", or
  "the spike's deliverable": an honest report of remaining work is legitimate ONLY as a
  preface to continuing it in the same turn.

Documentation-only (skill prose). Mirrors CLAUDE.md "Hard Cutover by Default" — the
Forbidden-excuse catalog, updated in the same cutover.
