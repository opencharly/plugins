# 2026.190.1039 — git-workflow: correct the self-approval + autoMode claims

Two claims in the B5 landing section were false. Both are corrected here.

**1. `autoMode.allow` does not take permission-rule syntax.** The prior text told
authors to put `Bash(gh pr merge:*)` into `autoMode.allow` "so the pr-validator
merges autonomously". That is a category error: `permissions.allow` holds
deterministic `Tool(pattern)` RULES, while `autoMode.{allow,soft_deny,hard_deny}`
holds the auto-mode CLASSIFIER's NATURAL-LANGUAGE policy statements (see
`claude auto-mode defaults`; every built-in entry is prose, none is a `Bash(...)`
pattern). Permission syntax placed there is inert. Verified live: with such entries
present, a sub-agent's status POST was denied twice.

**2. A commit status is agent-ATTESTED validation, not two-party review.** The prior
text presented "the COMMIT STATUS has no self-approval restriction" as the design's
justification. Claude Code's auto-mode classifier names this pattern directly — its
**Self-Approval** rule blocks "triggering a pipeline that marks the agent's own PR's
required checks as passed … regardless of whether the agent believes it verified its
own code", and a sub-agent the session spawned is "an automation the agent controls".
A fresh `pr-validator` gives CONTEXT independence (it catches real defects) but never
PARTY independence: same principal, same token, `required_approving_review_count: 0`.

The section now states the real mechanism: Self-Approval and Merge Without Review are
`[named+specifics]` SOFT blocks, clearable only by consent that NAMES the action; a
`<teammate-message>` never meets that bar; a standing operator rule in
`autoMode.allow` (natural language) does. `gh pr merge --auto` is explicitly carved
out of Merge Without Review on a protected repo.
