## 2026-07-03 — agents skill: the podman store-poisoning fix is the reap, not a cap/transient_store

Rewrote the bed-concurrency podman note to document THE fix (landed in
`charly/deploy_executor_nested.go`): a killed `podman run --rm` check-box probe
orphans its container (`--rm` fires only on exit), orphans poison podman's single
sqlite container-store → `database is locked` under concurrency. charly now names
each disposable probe container `charly-probe-<pid>-<seq>` and force-reaps it, so
a killed probe can never orphan. Podman itself is NOT concurrency-limited (32
concurrent builds on the real store, 0 failures); `transient_store` and
concurrency caps are disproven/unnecessary. Supersedes the earlier
"keep-the-store-clean-manually / transient_store escape hatch" framing.

Documentation-only.
