# 2026.190.1202 — autonomous landing: sub-agent project root is load-bearing for permissions

The autonomous PR-landing loop was silently broken for every submodule PR. Root cause,
established by controlled experiment rather than inference, and now encoded in the skills.

**What broke.** A `pr-validator` sub-agent's project root is its WORKING DIRECTORY, and
Claude Code loads `.claude/settings.json` (`permissions.allow` AND `autoMode.allow`) from
that root. Validators were being spawned with their working directory inside `plugins/` or
`sdk/` — submodules that ship no `.claude/` — so they loaded NEITHER. The classifier then
denied their `success` status POST as **Self-Approval**, reporting "the only authorization
comes from a `<teammate-message>`, which is not user intent": it had never seen the
standing rule that lives in the superproject's `.claude/settings.json`. Every downstream
conclusion ("the rule is inert",
"autoMode.allow takes the wrong syntax", "autonomy needs an operator in the loop") was
drawn from that broken setup.

**Proof (single variable, rule text unchanged).** Rooted in `plugins/`: `success` POST
DENIED. Rooted in the superproject, same agent, same rule: `success` POST **ALLOWED**, PR
merged, **zero denials**. Scope was the entire cause.

**Encoded now:**

- `/charly-internals:git-workflow` B5 gains **the autonomous-landing contract**: spawn every
  `pr-validator` rooted in the SUPERPROJECT, for a PR in ANY repo; drive submodules with
  literal `git -C /abs/path` + `gh --repo`; never `cd <submodule> &&`. B7 already stated
  this for the commit gate — it is equally load-bearing for PERMISSIONS. The loop is
  autonomous wherever an operator has committed that standing rule to the superproject's
  `.claude/settings.json` (a separate superproject change proposes committing it there, so
  a whole team inherits it).
- `/charly-internals:agents` gains **"Sub-agent operational invariants"**: project root =
  cwd; a permission denial ENDS the agent's turn (so verdicts are written durably BEFORE
  any gated action — a `failure` status/comment is never gated); reconnect via durable
  state (statuses, comments, `~/.claude/projects/<cwd-slug>/<uuid>.jsonl`) rather than the
  message channel; wait with a `run_in_background` until-loop whose exit condition covers
  EVERY terminal state, because silence is not success.
- `internals/agents/pr-validator.md` gains operating invariants W1–W3 so every future
  validator inherits the fix: stay in the superproject; record the verdict durably first;
  never route around a denial, and never post `success` on a PR you did not pass.

Do not "fix" a permission denial by editing the rule until the agent's project root is
confirmed.
