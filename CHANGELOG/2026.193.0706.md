# 2026.193.0706 — golangci-lint is now a pre-commit criterion (doctrine)

The `pre-commit-gate.sh` hook gains a Go-lint criterion: a commit staging a `*.go` change whose
touched MODULE is not `golangci-lint`-clean is BLOCKED. This is the doctrine half — the hook +
gate_test + CLAUDE.md enumeration land in the superproject commit that bumps to this plugins commit.

- **`internals/skills/agents/SKILL.md`** — the "Hooks doctrine" PreToolUse-gates enumeration gains
  the Go-lint criterion: the CONFIGURED `golangci-lint run` (never `--fix`/`--enable-only`) per
  touched module, `GOWORK=off` for `candy/plugin-*` candies, whole-MODULE analysis (because
  `unused` needs whole-package), fail-OPEN when golangci-lint is absent or times out (the
  `pr-validator` remains the real gate). Motivated by the P10 VM-CLI sweep's 21 orphaned symbols
  that the prior gate — which ran `go test` + the F11 gate but NOT the linter — let through.
- **`internals/skills/go-quality/SKILL.md`** — a note under "R10 gate by change class" that the
  pre-commit hook now enforces `golangci-lint`-clean at commit time, distinct from (not a
  replacement for) the bed R10 gate.

Rationale: golangci-lint's `unused` is exactly the linter that would have caught the P10 dead code;
making it a mechanical pre-commit backstop means dead/unused code can't land at all, on any Go
commit, without a human or validator having to remember to run it.
