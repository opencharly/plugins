## Member 1 — the collision catalog for mechanical type-repoint sweeps

A parallel session, doing a mechanical package-alias repoint sweep across
many files, repeatedly hit the same hazard class: a plain word-boundary
text/regex substitution cannot see Go's AST, so it silently mis-rewrites
method names, composite-literal field keys, shadowing locals, and field
declarations that happen to share the target symbol's spelling.

Added a new section, "Mechanical type-repoint sweeps — the collision
catalog", to `internals/skills/go-quality/SKILL.md` (between "Auto-fix
safety" and "gopls modernize" — both neighbors are about mechanical-rewrite
risk, so it reads as unsafe-blanket-autofix → this catalog → the one
mechanical rewrite tool that IS safe). Covers seven recurring collision
classes (method-name sharing, composite-literal field keys, local-variable
shadowing, the ellipsis-vs-selector-dot exclusion bug, tool self-exclusion
path-compare mismatches, generic-name amplification, and colon-less field
declarations) and a three-step mitigation ladder (proactive scan → an
AST-aware rewrite tool for the classes it genuinely covers → the compiler as
the non-optional final gate), plus the per-commit PR-body convention for
auditing a multi-symbol repoint commit-by-commit. Text finalized and
approved by the teammate who drafted it (T-54), committed here verbatim at
its specified insertion point.

### Round-1 fix — the mitigation ladder's own tool claim was unverified

The batch's round-1 validator ran a LIVE SPIKE against real `gofmt`
(go1.26.5) rather than trusting the shipped claim that `gofmt -r`
"inherently cannot touch... composite-literal keys, or local identifiers" —
and falsified HALF of it: `gofmt -r` genuinely cannot touch selector fields
or method names (an AST-node-kind guarantee), but it DOES wrongly rewrite
composite-literal field keys (class 2) and shadowed local identifiers
(class 3), producing invalid code in both cases — precisely because those
are bare identifiers indistinguishable from a genuine type reference at the
AST position the tool's pattern matches. The unverified claim traced back to
the orchestrator's own ruling to the drafting teammate, not to the
teammate's own tested method. Fixed in place: step 2 of the mitigation
ladder now scopes the AST-aware-rewrite guarantee to classes 1/4/5/7 only,
states the falsified classes 2/3 explicitly, and names the proactive scan
(step 1) as the PRIMARY defense for those two classes regardless of which
rewrite tool is used; step 3 (the compiler) is restated as mandatory
REGARDLESS OF REWRITE TOOL, since even the AST-aware tool can silently ship
invalid code for classes 2/3. The seven-class catalog itself was unaffected
— only the mitigation-ladder tool claim was wrong. The irony is exact: a
section teaching collision-catalog verification itself shipped an
unverified tool claim — the same claim-divergence class R1/R5 exist to
catch — and the process caught it before merge, via a live spike rather
than a second unverified assertion.

## Member 2 — validator evidence discipline (five session lessons)

Five distinct validation-round findings this session, all in the SAME
failure class (an unverified or ambiguous claim, or an unverified branch
state, acted on as if it were verified), motivated a new compact item in
`internals/skills/agents/SKILL.md`'s "Sub-agent operational invariants" list:

- A plugins CHANGELOG's unqualified "PR #126" was flagged as a fabricated
  citation in PR #70's round-1 validation, because the validator checked
  the wrong repo first — bare cross-repo `#N` citations are ambiguous the
  instant they're read from a different repo than the one they name; the
  fix is always `owner/repo#N`.
- The repo set a citation check must search is `opencharly/{charly, plugins,
  sdk, distro-*, pkg-*, .github}` — checking only the CHANGELOG's own repo
  (or guessing) is how a real citation gets misdiagnosed as fabricated.
- Three independent validators this session verified a claim against a
  LOCAL checkout (`/home/atrawog/Atrapub/o/charly`) without checking whether
  it had fallen behind `origin/main` — a local checkout must be treated as
  potentially stale, always cross-checked against the remote.
- Evidence pasted into a PR body or CHANGELOG (a grep count, a checklist
  line, a balance check) must be RE-RUN at the final head immediately
  before pasting, never transcribed from memory or an earlier run — PR
  #70's own Finding B (a grep count that didn't reproduce) and a parallel
  sdk-repo validator round hit the identical mistake.
- **The fifth lesson, discovered assembling THIS very batch:** PR #71
  (this batch's own predecessor) was validated and merged (squash `a11d933`,
  branch deleted) while a hold-comment request was still in flight to it —
  a genuine race, not a mistake by either side. The teammate carrying rider
  work for #71 then ran its normal `git fetch` before pushing a follow-up
  commit; the fetch reported "couldn't find remote ref" for a branch it had
  pushed to minutes earlier. That message was read as "nothing to
  reconcile" and the push proceeded — silently RECREATING an orphan branch
  of the same name, now detached from the merged (and closed) PR. The
  correct read: a missing remote ref for a branch you believe exists means
  its state changed under you — re-derive the real state (check the PR)
  BEFORE pushing again, never push through the anomaly. No damage resulted
  (nothing force-pushed, no PR affected, the orphan was later confirmed
  unreferenced and cleanly deleted), but the near-miss is exactly the class
  this item's other four lessons already cover — an unverified assumption
  acted on instead of checked.

## Member 4 — post-commit staging verification

Added a bullet to `internals/skills/git-workflow/SKILL.md`'s non-negotiable
invariants, beside the existing "Right worktree" bullet (the same class of
silent-failure mode): after EVERY commit, re-run `git status --short` AND
`git show --stat` to confirm the commit actually contains every intended
file — a multi-path `git add` where one pathspec is mistyped or stale can
stage only the files that DID resolve while `git commit` still succeeds,
producing an incomplete commit that would not even compile. This session
caught exactly such an 8-file commit only by habitually re-checking
`git show --stat`, not by any tooling that would have failed loudly on its
own. (This very rule was applied to this batch's own assembly commit before
pushing — see the gate evidence in the PR body.)

## Member 5 — the TRACE SPIKE for relocation/fold cutovers

Added one sentence to `internals/skills/strict-policy/SKILL.md`'s RDD
"spike" paragraph, naming a new spike variant alongside the existing SCHEMA
