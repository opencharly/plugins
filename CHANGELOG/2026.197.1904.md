## Member 1 — the named boundary-law trap (`internals/skills/plugin/SKILL.md`)

Two operator-run architecture auditors independently mis-classified the
deploy and check cones as ~90% permanently-core, both using the same excuse:
a construct "can't cross the process boundary" or "drives podman/ssh/flock/
systemd itself," so it must stay in `charly/`. The true kernel in those
cones turned out to be ~0.3%. What let one of them self-correct was a
concrete grep, not the abstract E/M/B/D test alone. Two more auditors
separately self-corrected on the (E) bucket specifically — overclassifying
genuine residue and a concrete kind's typed shape as permanent envelopes.
Added five new paragraphs to "The kernel/plugin boundary law" section,
immediately after the existing "un-gameable self-test":

- **The named trap** — calling a construct a "host-boundary object" is NEVER
  a permanence reason; a plugin drives that same host object itself (as
  `candy/plugin-vm` and `candy/plugin-kube` already do) or reaches a live
  venue over the reverse-channel broker (M4) — needing the broker or the
  host IS the seam's whole point, never a reason to stay core.
- **The defines-vs-calls test** — the concrete grep that operationalizes the
  trap: grep for where a mechanism is DEFINED (`provider_registry.go`,
  `materialize.go`, the reverse-channel broker in `plugin_inproc*.go` /
  `plugin_dispatch_reverse.go`, all grep-verified live in `charly/` before
  citing); a file that only CALLS one of those is a capability that USES the
  mechanism, not the mechanism itself — an R-item, it moves. "A capability
  that uses M1–M4 is not thereby M1–M4."
- **Alias-is-always-residue** — a `type X = spec.X` re-export in `charly/`
  is ALWAYS an R-item under ZERO-ALIASES, regardless of what it aliases,
  even a genuine E-envelope type — the alias IS the mislocated call site.
  Grep-verified example: `labels.go`'s `BoxMetadata = spec.BoxMetadata` and
  `install_plan.go`'s `Scope`/`Venue`/`Phase` aliases, all genuine `spec.*`
  E-types, all still R-items an auditor initially miscounted as permanent.
- **A concrete kind's typed shape is R, not E** — the (E) bucket is ONLY
  kind-AGNOSTIC carriers; a struct named after a concrete kind with
  kind-specific accessors (`Candy`, `VmSpec`, `ResolvedBox`) is that kind's
  typed shape, an R-item. Grep-verified example: `layers.go`'s `Candy`
  struct (60+ accessor methods, every one kind-specific) was initially
  miscounted as E because it "carries data" — carrying data is not the E
  test, kind-agnosticism is.
- **The practical audit framing** — invert the default: every file is an
  R-item (moves to a plugin) UNLESS it is literally one of the tiny kernel
  whitelist (the four in-core M-mechanisms, the B bootstrap root, D
  kind-recognition data, or an E generic envelope). Names the bed runner,
  the check/ADE harness, the build engine, the deploy walk, and the VM
  engine as R-items by this test, and cites the ~90%-vs-~0.3% incident as
  the motivating example of the failure this framing forecloses.

## Member 2 — generalize-aggressively (`internals/skills/plugin/SKILL.md`)

A second insight from the same audit round: a K-wave move that just
relocates a per-case branch verbatim (rather than collapsing it into a
generic mechanism) under-delivers on the boundary law's own intent. Added a
paragraph immediately after "The canonical shape every kind copies —

## Member 4 — orchestration discipline (`internals/skills/agents/SKILL.md`)

Three operator-run disciplines proven live this session, added as a new
"Brief verification + stop-and-respawn — correct the teammate, don't
misdirect it" section immediately before "The responsibility matrix":

- **Verify a brief's criterion against CLAUDE.md + the owning skill BEFORE
  dispatch** — a wrong brief misdirects every teammate it reaches at once.
  Motivating incident: an audit brief encoding a boundary-law violation
  ("host-boundary objects stay permanent") sent all 3 auditors to an
  inverted verdict.
- **STOP-AND-RESPAWN over repeated in-place correction, only once CONFIRMED
  still wrong.** A teammate that re-applies a wrong approach after one clear
  correction, or accumulates same-class errors across rounds, is stopped and
  replaced with a fresh teammate on a clean brief — but first confirm it
  hasn't already self-corrected, since an async correction can cross a
  teammate's own in-transit self-correction message (the same collision
  "Crossed-ruling reconciliation" describes for rulings). A respawn already
  issued when a valid self-correction arrives is ACCEPTED as an independent
  cross-check, never discarded as wasted. Motivating incident: an auditor
  self-corrected a check-cone mis-classification structurally, its message
  crossing the orchestrator's respawn — the respawn became a useful second
  pass.
- **Whack-a-mole escalation** — a recurring same-error-CLASS across
  validation rounds escalates to a root-cause fix or a fresh clean-pass
  agent, never another single-item patch. Cross-references (rather than
  re-narrates) Member 3's alias-residue precedent (5 rounds → one
  root-cause de-fragilization).

CLAUDE.md's "Agents, Workflows & Teams" section gets a matching one-line
mandate (brief verification + stop-and-respawn) with an extended `*Detail:*`
pointer — mandate in CLAUDE.md, full framing here (R3).

## Member 5 — git-mv stale-pathspec CalVer-finalization guard (`internals/skills/git-workflow/SKILL.md`)

A third recurrence of the same footgun this session: `git mv
CHANGELOG/<placeholder>.md CHANGELOG/$VER.md` stages the rename, but a
follow-up `git add <old-path> <new-path>` naming the now-nonexistent old
path FAILS and silently drops staging any content edit made in the same
step (the H1 heading rewrite) — a rename-only commit that misses the edit.
Added a bullet to "CalVer — generated AND rewritten at MERGE" (the
evaluator's merge-time step list): verify `git show HEAD:<new-path> | head
-1` (or `git diff --cached`) matches the intended content BEFORE committing
or posting the status. Recurred in `charly` PR #131, `pkg-arch` PR #4, and
the fresh evaluator finalizing `plugins` PR #75's own CalVer — the third
occurrence is what this guard is named for, caught pre-push only because
the self-verify was actually run.

`internals/agents/pr-validator.md`'s Phase 3 (finalize sequence) already
carries the equivalent SELF-VERIFY instruction (`head -1 CHANGELOG/$VER.md`
byte-equal check before posting the final status) — verified present and
accurate; no further edit needed there, so this member touches only the
skill (R3 — no duplicate instruction added where an accurate one already
exists).
