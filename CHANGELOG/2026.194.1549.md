# 2026.194.1549 — codify five wave-2 orchestration/landing insights into the skills

Documentation-only cutover folding five PROVEN, generalizable insights from the
wave-2 core-minimization program into the committed skills (the operator's
"constantly update skills with generalizable insights" directive). Each insight
is stated present-tense and generic in the owning skill; the paired superproject
pointer bump rides the same logical cutover.

**`internals/skills/git-workflow/SKILL.md`:**

- New subsection **"The cross-repo WIP landing sequence — commit-to-rebase without
  shipping unproven code"** (beside the B1 branch loop): when a multi-repo cutover's
  WIP must be committed to rebase (a submodule-spanning tree makes `git stash`
  unsafe — a gitlink stash can drop an in-progress pointer) but the runtime-class
  commit gate needs live proof of the FINAL code that only exists after the rebase,
  the sequence is (a) freeze + exploratory roster, (b) commit LOCALLY at the
  then-honest `analysed on a live system`, do NOT push, (c) rebase onto the current
  mains + regenerate every generated file (never hand-merge), (d) re-run the gates +
  re-freeze, (e) the FINAL roster runs on the rebased final code, (f) amend/reword
  to the EARNED tier — legal only because the branch is UNPUSHED — then push + open
  the PRs. Nothing pre-bed ever ships; the tier only ever moves UP.

**`internals/skills/agents/SKILL.md`:**

- The bed-scoped parallel-testing "Per-worktree binaries" item (4e) gains the
  **CalVer-stamp requirement**: a per-worktree bed binary MUST be built with
  `task build:binary` (`-ldflags "-X main.BuildCalVer=<calver>"`), NEVER a bare
  `go build -o` — an unstamped binary reports version `unknown` and fails every bed
  step asserting the CalVer stamp (a `vm:` bed pushes the host binary into the guest
  and asserts `charly version` there). The scheduling rule also gains: verify every
  bed NAME against the LIVE tree roster (`grep '^check-.*:' charly.yml`) before each
  launch, because rosters change across cutovers and a stale name aborts the run.
- New subsection **"Crossed-ruling reconciliation — one unambiguous instruction
  beats several partial ones"** (beside the north-star protocol): orchestrator
  rulings and teammate checkpoints cross in flight, so a ruling that may cross an
  in-flight report is followed by a NEXT message that explicitly reconciles both
  states ("my X superseded / stands; your Y applies / reverts"). An unreconciled
  pair of partial rulings can make a teammate revert R10-gated work; a teammate
  receiving apparently-conflicting rulings STOPS-and-asks rather than picking one.

**`internals/skills/cutover-policy/SKILL.md`:**

- New required-deliverable bullet **"Relocation step-sequence PARITY"** (for a
  cutover relocating a dispatcher / orchestrator / runner): enumerate the original's
  per-CLASS behavior arms (bring-up / update / teardown × each substrate or shape
  class) before relocating, and prove each class's step sequence matches in the
  relocated code. The characteristic relocation defect is fixing one arm of a class
  while missing its siblings — a generic fallback that silently mis-handles a special
  class passes every unit test and fails only on the class's live bed — so the
  acceptance roster must carry a bed per SPECIAL class.

**`check/skills/check/SKILL.md`:**

- The exit-code discriminator note gains one line: a wrapper script's own exit code
  is NOT the bed verdict — a script that pipes or post-processes `charly check run`
  must capture `${PIPESTATUS[0]}` (the charly exit) and corroborate the on-disk
  `summary.yml`; a notification-level exit 0 can mask a charly exit 1.
