## 2026-07-03 — agents skill: the empirical CPU-oversaturation strategy for multi-concurrency bed runs

Added the measured concurrency model to the bed-fan-out section. CPU load is the ONE
real ceiling — the store (reap) + DNS (reconcile) fixes hold under any concurrency and
RAM/disk have huge headroom; what degrades is CPU, because every candy compile step
runs `make -j$(nproc)` INSIDE its container (the dominant, charly-invisible multiplier),
so N beds mid-build ≈ N×nproc threads. Measured: maxjobs 8 + 3 VMs peaked at load 42
(2.6× cores) and SIGKILLed check-live probes on jupyter/openclaw/sway — the same beds
pass at maxjobs 3. The optimizing limit is LOAD-GATED admission (hold new BUILD starts
while 1-min load ≥ cores), not a fixed maxjobs, since phase-diverse beds (pull/deploy/
check burn ~no CPU) overlap a compiling bed for free. Also: a build that FAILS under
load is almost always a deterministic bug (isolate by rebuilding ALONE first — RDD),
not the concurrency. Documentation-only.
