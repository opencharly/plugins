## What was stale, and what's actually true now

`internals/skills/install-plan/SKILL.md` described the pod-overlay build-emit
dispatch as a host-only mechanism: `ociEmitStep` directly resolving the
`class:step` provider via `providerRegistry` and Invoking its `OpEmit` via the
shared `spliceClassStepEmit`/`invokeOpEmitFragment` seam, with host-coupled
step words (`system-packages`/`builder`/`local-pkg-install`/`op`) additionally
described (in places) as still calling back a host `step-emit` renderer
(`stepEmitSystemPackages`/`stepEmitBuilder`/`stepEmitLocalPkgInstall`/`stepEmitOp`)
registered via `registerStepEmitter` — a description that predates the
K5-Unit-6b relocation of those four renderers directly into
`candy/plugin-installstep` (against its own `"resolved-project"`-built
`deploykit.Generator`, no host round-trip), and now additionally predates
K5-A item 2's relocation of the DISPATCH DECISION itself off the host
entirely.

Every table row and prose passage in `internals/skills/install-plan/SKILL.md`
naming `ociEmitStep`'s host-side registry consult, `ociSpliceClassStepEmit`/
`spliceClassStepEmit` (both now-deleted identifiers), or the stale
host-coupled-renderer callback path is corrected to describe the current
mechanism: the host's `charly/oci_step_emit.go` is a thin forwarder over
`HostBuild("step-emit","oci-emit-step")`; `candy/plugin-installstep`'s
`"oci-dispatch"` word does the FULL dispatch (external-step / the 12
compiler-emitted kinds / the `ExternalPlugin` verb step) via `DescribeProvider`
(cached `StepContract` metadata, no live Invoke) + `InvokeProvider` (dispatch);
the four host-coupled kinds render directly against the plugin's own
`"resolved-project"`-built `deploykit.Generator`, no host `step-emit` callback.
