# 2026.193.1851 — build-family skills: the box-build DRIVE now lives in candy/plugin-build (P8b docs)

R5 claim-keyed sweep of the stale "the box-build engine STAYS host-side behind HostBuild(image/
containerfiles)" claim across every build-family skill, now that P8b + #24 landed (the box-build
podman DRIVE moved into candy/plugin-build; the host is a resolve/render seam provider). The NEW
standing truth, forward-looking:

- **`/charly-build:build`** + **`/charly-build:generate`** ("Build-engine dispatch") — the compiled-in
  `build:box`/`build:generate` plugin OWNS the podman DRIVE (build-order loop, per-image lock,
  `podman build`, push, inline-merge gate) IN the candy (sdk-only); it reaches the host loader
  RESOLVE + Containerfile RENDER + privileged builder-bootstrap via `HostBuild("build-resolve")`
  (→ a serializable drive-model) and the layer merge via `HostBuild("merge")`. `box generate` is
  `build:generate` → `HostBuild("build-resolve")` with `GenerateOnly` (host renders `.build/`,
  returns the written paths; no podman, no merge). The RESOLVE/RENDER stays host-side.
- **`/charly-internals:plugin`** (the `build` class + the HostBuild-kinds list) — the plugin's Invoke
  OWNS the drive; the registered kinds are `build-resolve` + `merge` (not `image`/`containerfiles`,
  removed); the RESOLVE/RENDER engine stays core (pinned by the loader Mechanism + the runtime-Candy
  decision), the podman DRIVE moved to the candy.
- **`/charly-internals:install-plan`** (`plugin_dispatch_reverse.go` row + the pod-overlay + step-emit
  sibling refs) — the box-build host-builder is `build-resolve`/`merge`; `overlay`/`step-emit` are its
  build-context siblings.

Documentation-only (no behavior change). Consumer: superproject sdk/plugins pointer bump.
