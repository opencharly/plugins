# 2026.197.1328 — Validator handoff

The PR-validator policy receives a complete parent handoff as transient spawn context,
never as an author-worktree artifact: the target protected base and PR head,
protected-policy object, repository/gitlink map, approval categories, and mutation
limits. A fresh validator uses that handoff to load protected policy and skills before
candidate actions, treating an unavailable required object, capability, or declared
gitlink as a durable `BLOCKED` result.

The validator does not bootstrap an additional checkout or rely on a helper task,
`/tmp` ledger, user-level setup, or a guessed relationship between a submodule's target
base and the superproject gitlink. Its phase-one verdict and any later denied-action
outcome are recorded durably as PR comments before any gated action; it streams those
comments and the squash body without creating validator-local files. Validator and
workflow attribution now use the common model-aware `Assisted-by:` syntax.
