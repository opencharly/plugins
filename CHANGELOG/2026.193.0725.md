# 2026.193.0725 — sub-agents READ skills by path (the reliable method); Skill-by-name is only a shortcut

Fix for the recurring sub-agent failure "The named skills aren't available as invocable skills in
this environment (they're documentation skills referenced by CLAUDE.md, not registered in the Skill
tool here)." Root cause (RCA, 2026-07-12): the charly-* skills are registered by `enabledPlugins` +
`extraKnownMarketplaces` in the SUPERPROJECT `.claude/settings.json`, and a sub-agent's `Skill`-tool
registration of them is INCONSISTENT — the tool can be present (the "Option A" cutover added it to
every roster agent's `tools:`) with ZERO charly-* skills registered. The Option A guidance to
"invoke skills BY NAME" therefore fails in those contexts, and the agent rationalizes the skills as
"just documentation" instead of Reading the file. The reliable method — a plain file read of
`SKILL.md` by PATH — has no marketplace dependency and always works.

- **`internals/agents/pr-validator.md`** — the "R0 — load your PR-validation skills FIRST" section
  is rewritten: **`Read` each `SKILL.md` by PATH** is now the PRIMARY instruction (with the concrete
  `plugins/<family>/skills/<name>/SKILL.md` paths); `Skill(name)` is an optional shortcut, and a
  "not registered" result is EXPECTED — `Read` the file, NEVER conclude "skills unavailable".
- **`internals/skills/agents/SKILL.md`** — a new "Sub-agent operational invariants" #4 documents the
  mechanism (the marketplace lives in the superproject `.claude/settings.json`, same file invariant
  #1 governs) and the rule: Read-by-path first, Skill-by-name a shortcut, never treat "not
  registered" as "absent".

Pairs with the CLAUDE.md R0 sub-agent note (superproject commit that bumps to this plugins commit).
Corrects the earlier same-session over-generalization that a superproject-rooted probe's successful
by-name invocation meant "the roster agents load BY NAME" — a fragile default; Read-by-path is the
invariant that holds.
