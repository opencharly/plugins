# 2026.190.1717 — hooks-doctrine: document the fail-closed-on-unparseable gate behaviour

Paired documentation for the superproject cutover that rewrites the PreToolUse gates onto
a shared `shlex` tokenizer with no fallback (`.claude/hooks/gitcmd.py` + both gates).

The `/charly-internals:agents` hooks-doctrine already correctly states that the gate's
absent-trailer check does not scan a `-F`/heredoc/command-substituted message for absence
(tier legality still applies). That sentence is KEPT — it is true: a clean heredoc /
`-F <file>` message tokenizes and is judged normally, not scanned for absence.

What is NEW under the tokenizer is a fail-closed behaviour, added to the same sentence: a
command the gate cannot **tokenize** — an **unbalanced or unquoted quote** (e.g. an
apostrophe in a heredoc body, or an unterminated `"`) — fails CLOSED and is blocked; the
remedy is to balance the quotes or use `git commit -F <file>`.

(An earlier draft of this cutover wrongly said "a heredoc is blocked" and replaced the
correct sentence; that was refuted by the paired PR's own validator — a clean heredoc
tokenizes and is allowed — and is not carried here. R1: the doctrine matches observed
behaviour.)
