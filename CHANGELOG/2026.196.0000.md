# 2026.196.0000 — The Cutover Sizing Law (batch cutovers codified)

The operator standing directive to batch non-blocking fixes into large
thematic cutovers — issued repeatedly in-session but never codified — kept
losing to the committed per-issue default ("a non-blocking issue as its OWN
immediate-next cutover"), which every hook reminder and fresh context
re-derived. Root cause: an uncodified standing order loses to a codified
contrary default. This cutover makes the batch the codified meaning of
"immediate-next cutover" for non-blocking work.

- `cutover-policy` gains **"Cutover sizing — the batch law (LARGE by
  default)"**: the ceremony-cost rationale (fixed hours per landing,
  measured ~85% overhead on sub-800-line solo cutovers), thematic batches,
  the batch-vs-deferral distinction (a batch is planned/owned/begun NOW),
  the solo-ceremony exceptions (blocking/urgent, or a substantial ≥~1.5k-line
  coherent scope), gate-roster trimming (run only the beds that can FAIL on
  the change), and concurrent independent validator legs.
- `strict-policy` R2 path 3 and its doc-divergence worked example now route
  non-blocking work to the named thematic batch.
- `agents` responsibility matrix: the orchestrator OWNS the thematic batch
  queues; a small-fix teammate brief NAMES its batch; a solo ceremony for a
  small non-blocking fix is a routing error.
- `git-workflow` stray-artifact note + `pr-validator` items 6/13 now demand
  a NAMED batch/task, never a vague "follow-up".
- Companion superproject change: CLAUDE.md R2 + "Hard Cutover by Default"
  (the Cutover Sizing Law mandate) + the post-execution paragraph + the
  end-of-turn hook line.
