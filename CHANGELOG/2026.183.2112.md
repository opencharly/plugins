## 2026-07-02 — Skills: podman store-lock concurrency + below-floor per-host config gotchas

Captured two proven operational learnings from a full concurrent eval-bed roster run:

- **`/charly-internals:agents`** (bed-scoped parallel section): the one concrete
  concurrency ceiling when fanning the roster out is podman's **sqlite
  container-store write-lock** — every container CREATE (`podman build` +
  `podman run --rm`) serializes on it. Oversaturating one busy store (many
  concurrent bed builds + external `podman` commands + orphaned containers) fails
  ops with `database is locked` (rc 125). A `podman run --rm` killed mid-create
  orphans a never-removed `Created` container; these accumulate into a spiral.
  Measured: podman handles reasonable concurrency fine on a CLEAN store (48
  concurrent creates ~2s; 16 on a large real store ~27s, 0 failures) — so keep the
  store clean (sweep anonymous `Created` orphans) and don't pile external `podman`
  load onto a running roster. `transient_store` is podman's escape hatch for
  extreme churn but is usually unnecessary.
- **`/charly-build:migrate`** (below-floor section): a below-floor **per-host**
  config (`~/.config/charly/charly.yml`) blocks not just LOADS but DEPLOYS —
  `saveDeployState` refuses to overwrite a config it cannot load, so every
  pod/vm/local deploy (incl. `charly check run <bed>` at its config step) fails
  until the per-host config is reset (it holds only regenerable deploy state).

Documentation-only.
